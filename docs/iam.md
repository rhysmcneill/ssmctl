# IAM Reference

This page lists the exact IAM permissions required for each `ssmctl` command, and provides copy-paste policy fragments you can attach to your IAM identities or instance profiles.

All permissions are on the caller's IAM identity (the person or role running `ssmctl`) unless noted as an **instance role** permission.

---

## Minimum policy for all commands

This policy covers `list`, `connect`, `run`, `cp`, and `forward` with no S3 staging:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "SSMCoreAccess",
      "Effect": "Allow",
      "Action": [
        "ssm:DescribeInstanceInformation",
        "ssm:StartSession",
        "ssm:TerminateSession",
        "ssm:SendCommand",
        "ssm:GetCommandInvocation"
      ],
      "Resource": "*"
    },
    {
      "Sid": "EC2DescribeInstances",
      "Effect": "Allow",
      "Action": [
        "ec2:DescribeInstances"
      ],
      "Resource": "*"
    }
  ]
}
```

> **Tip:** Replace `"Resource": "*"` with specific instance ARNs or tags to tighten the policy for production use. See [Restricting access by target](#restricting-access-by-target).

## Per-command breakdown

### `ssmctl list`

```
ssm:DescribeInstanceInformation
ec2:DescribeInstances
```

### `ssmctl connect`

```
ssm:StartSession
ssm:TerminateSession
ec2:DescribeInstances          (for Name tag resolution)
```

### `ssmctl forward`

```
ssm:StartSession
ssm:TerminateSession
ec2:DescribeInstances          (for Name tag resolution)
```

### `ssmctl run`

```
ssm:SendCommand                (document: AWS-RunShellScript)
ssm:GetCommandInvocation
ec2:DescribeInstances          (for Name tag resolution)
```

### `ssmctl cp` (in-band, no S3)

```
ssm:SendCommand                (document: AWS-RunShellScript)
ssm:GetCommandInvocation
ec2:DescribeInstances          (for Name tag resolution)
```

### `ssmctl cp --via s3://...`

The S3-backed transfer path requires permissions on **both** the caller's identity and the **instance role**:

**Caller identity (the person running ssmctl):**

```
ssm:SendCommand                (document: AWS-RunShellScript)
ssm:GetCommandInvocation
ec2:DescribeInstances
s3:PutObject                   (upload: caller stages the file)
s3:GetObject                   (download: caller fetches the staged file)
s3:DeleteObject                (cleanup of staging object)
```

**Instance role (the EC2 instance's IAM role):**

```
s3:GetObject                   (upload: instance pulls from staging)
s3:PutObject                   (download: instance pushes to staging)
```

## cp via S3

Paste this policy onto the **caller's** IAM identity. Tighten the `Resource` to your actual bucket and prefix.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "SSMCtlS3Staging",
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetObject",
        "s3:DeleteObject"
      ],
      "Resource": "arn:aws:s3:::my-staging-bucket/ssmctl/*"
    }
  ]
}
```

Paste this policy onto the **instance role** (the role your EC2 instance assumes):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "SSMCtlInstanceS3Access",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::my-staging-bucket/ssmctl/*"
    }
  ]
}
```

The prefix `ssmctl/*` matches the keys generated by `ssmctl` (`ssmctl-<random>-<filename>`). You can be more permissive (`*`) or more restrictive (a fixed prefix) depending on your security posture.

## Restricting access by target

To restrict `connect`, `forward`, `run`, and `cp` to specific instances, scope `ssm:StartSession` and `ssm:SendCommand` by instance tag or ARN:

**By instance tag:**

```json
{
  "Sid": "SSMAccessToTaggedInstances",
  "Effect": "Allow",
  "Action": [
    "ssm:StartSession",
    "ssm:SendCommand"
  ],
  "Resource": "arn:aws:ec2:eu-west-1:123456789012:instance/*",
  "Condition": {
    "StringEquals": {
      "ssm:resourceTag/Environment": "production"
    }
  }
}
```

**By specific instance IDs:**

```json
{
  "Sid": "SSMAccessToSpecificInstances",
  "Effect": "Allow",
  "Action": [
    "ssm:StartSession",
    "ssm:SendCommand"
  ],
  "Resource": [
    "arn:aws:ec2:eu-west-1:123456789012:instance/i-0123456789abcdef0",
    "arn:aws:ec2:eu-west-1:123456789012:instance/i-0987654321fedcba0"
  ]
}
```

## Instance role — SSM Agent baseline

For an EC2 instance to be reachable via SSM at all, its instance role must include the AWS managed policy:

```
arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore
```

Attach this to your instance role. It grants the SSM Agent the permissions it needs to register with the SSM service, receive commands, and establish sessions.

## Checking your effective permissions

If a command fails with an `AccessDenied` error, use the AWS CLI to check what policies are attached to your identity:

```bash
# Who am I?
aws sts get-caller-identity

# What policies are attached?
aws iam list-attached-user-policies --user-name <your-username>

# Simulate a specific action
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:user/your-username \
  --action-names ssm:StartSession \
  --resource-arns "*"
```
